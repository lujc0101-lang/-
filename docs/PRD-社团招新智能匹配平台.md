|------|------|
| 文档版本 | v1.0 |

---

## 1. 背景与目标

### 1.1 背景

高校社团招新存在：信息不对称、报名流程分散、新生难以快速找到兴趣契合的社团等问题。

### 1.2 产品目标

- **新生**：短路径发现社团 → 理解差异 → 获得可解释的智能推荐 → 一站式报名 → 跟踪状态。
- **社长/招新负责人**：标准化展示社团 → 配置招新信息与报名表 → 管理报名名单与状态 → 查看基础转化数据。
- **平台管理员（可选角色）**：审核社团入驻、发布公告与招新专场、必要时治理内容。

### 1.3 成功指标（建议，可按学期迭代）

| 指标 | 说明 | 备注 |
|------|------|------|
| 报名完成率 | 进入报名表单的用户中完成提交占比 | P0 需可统计 |
| 推荐点击率 | 推荐列表卡片点击占比 | P0 需可统计 |
| 社长处理时效 | 报名状态首次更新时间分布 | P1 |
| 用户满意度 | 学期末问卷 NPS/CSAT | 运营向 |

## 2. 用户与角色

### 2.1 角色定义

| 角色 | 标识 | 权限概要 |
|------|------|----------|
| 访客 | 未登录 | 浏览公开社团目录与详情；不可提交报名、不可收藏（或仅本地收藏，见 8.2） |
| 新生 | `student` | 问卷匹配、收藏/对比、提交报名、查看「我的报名」状态；维护个人画像（标签） |
| 社长 | `leader` | 绑定一个或多个社团（见数据模型）；编辑社团资料与报名表、处理报名、查看社团数据看板 |
| 管理员 | `admin` | 审核社团、全站公告、招新专场配置、用户角色分配（最小实现可先做「邀请码 + 首名管理员种子」） |

### 2.2 假设

- 同一用户可**仅为一种业务角色**（一期简化：`student` 与 `leader` 分身账号二选一；若同学兼任，用两个账号或二期合并身份）。
- 社长绑定社团需**可验证**（一期：`admin` 审核 或 `admin` 生成的邀请码完成绑定）。

## 3. 范围与分期

### 3.1 总范围（本次 PRD 覆盖）

包含先前讨论的全部能力，分为 **P0（一期必交付）**、**P1（二期增强）**、**P2（差异化/运营深化）**。研发排期以 P0 → P1 → P2 为默认顺序。

### 3.2 P0 — 最小可用产品（MVP）

1. 社团目录（分类、搜索、排序、招新状态筛选）。
2. 社团详情（标准化字段 + 富文本/图文介绍）。
3. 智能匹配 MVP：固定问卷 → 规则打分 → Top N 推荐 + **每项推荐理由（可解释）**。
4. 报名表：协议定的「通用字段」+ 每社团「自定义题目（上限 configurable）」；提交后状态机（见 5.5）。
5. 账号体系：`student` / `leader` 注册登录（邮箱+密码 或 学号+密码，具体见技术选型；PRD 要求**可唯一标识用户**）。
6. 社长端（Web）：社团资料维护、报名表配置、报名列表与状态变更。

### 3.3 P1 — 体验增强

1. 收藏与对比（2～3 个社团并列对比字段差异与时间冲突提示）。
2. 站内通知：招新截止提醒、报名状态变更通知（站内信箱/铃铛，邮件可选）。
3. 招新专场页：按活动日聚合展示（如「百团大战」），支持配置展位号/地图图片 URL（不做真实地图 SDK 也可）。

### 3.4 P2 — 差异化与运营

1. 对话式匹配：自然语言输入 → 抽取结构化偏好 → 接入与问卷相同的打分引擎（可并存）。
2. 运营看板（管理员）：全站曝光、CTR、报名漏斗、各社转化排行。
3. 社团联办活动展示：多社关联一条活动，出现在多社详情/专场流中。

## 5. 功能需求详述

以下每条均含 **描述**、**规则**、**验收标准（AC）**。

### 5.1 社团目录（P0）

**描述**  
展示所有「已发布且审核通过」的社团卡片流/表；支持检索与排序。

**规则**

- 社团卡片字段：名称、一句话 slogan、封面图、分类、标签（最多展示 3～5 个）、`recruiting` 招新中标记、报名人数/浏览量（浏览量 P0 可做简单 PV）。
- 分类：预设枚举（文体、学术、公益、创新创业、其他）+ 管理员可配置扩展（P1；P0 可仅用固定枚举）。
- 搜索：对名称、slogan、标签、介绍全文（可二期）做模糊匹配；P0 至少名称+标签。
- 排序：默认「招新中的靠前」；可选「最热（近 7 日 PV）」「最新入驻」「名称 A-Z」。
- 筛选：仅招新中 / 全量。

**验收标准**

- AC1：招新中筛选与默认排序符合规则。
- AC2：搜索无结果时展示空状态与清空条件入口。
- AC3：未登录用户可浏览目录与进入详情。

---

### 5.2 社团详情（P0）

**描述**  
单社完整信息页 + 行动点（报名、收藏 P1、分享）。

**字段（P0 必填/选填）**

| 字段 | 必填 | 说明 |
|------|------|------|
| 名称 | ✓ | 唯一性由管理员/系统校验 |
| slogan | ✓ | 短文本 |
| 封面/相册 | 选 | 至少封面；相册可有 |
| 分类 | ✓ | |
| 标签 | ✓ | 多选，来自系统词表 |
| 详细介绍 | ✓ | 富文本或 Markdown |
| 日常活动类型 | ✓ | 枚举/多选 |
| 参与强度 | ✓ | 轻/中/重（或 1～5 星） |
| 招新时间段 | ✓ | 开始、结束（ISO 日期）；未開始/進行中/已结束自动计算 |
| 招新人数/说明 | 选 | |
| 报名表额外的对外说明 | 选 | |
| 外联方式 | 选 | 邮箱、QQ群号、微信群关键词等；**二维码图片**若学校允许则上传（一期可做 URL 占位） |
| 社长对外展示名 | ✓ | 可非公开展示手机号（默认脱敏） |

**验收标准**

- AC1：招新未开始/进行中/已结束在详情页明确展示。
- AC2：已结束招新时，「报名」按钮禁用并提示原因。
- AC3：PV 在详情页加载时 +1（防刷可做简单节流：同会话同社 30 分钟算一次，P1 细化）。

---

### 5.3 智能匹配（P0）

**描述**  
用户完成固定问卷 → 系统按 **可配置权重规则** 为每个社团计算匹配分 → 输出 Top N（默认 10）列表，每条带 **「匹配理由」**（人类可读条目列表）。

**问卷（建议默认题库，管理员可在后台改文案与权重，P0 可写死配置于配置文件）**

| 题目 ID | 题型 | 选项示例 | 映射到社团侧 |
|---------|------|----------|----------------|
| Q1 兴趣领域 | 多选 | 文艺、体育、科技、志愿服务、学术、社会实践… | 与社团 `tags` / `category` 命中加分 |
| Q2 可支配时间 | 单选 | ≤3h/周、3～6h、6～10h、10h+ | 与社团 `intensity` 相容性评分 |
| Q3 偏好活动 | 单选 | 表演/比赛、训练提升、社交松弛、项目产出… | 与 `activityTypes` 命中 |
| Q4 接受培训/例会频率 | 单选 | 低/中/高 | 与强度与社团描述关键词（可选） |
| Q5 是否零基础友好 | 布尔 | 是/否 | 社团字段 `beginnerFriendly` |

**计分规则（P0 逻辑，须实现为独立模块便于单测）**

- 基础分 0～100：各题加权求和；权重可配置表 `matchWeights`。
- **硬性过滤（可选）**：若招新已结束，则不进入推荐列表（或降权至末尾并标明）。
- **多样性**：Top 10 中同一一级分类默认不超过 4 个（可配置），避免刷同质。
- **理由生成**：每一题命中则追加一条理由，例如「你在兴趣中选择了「科技」，与该社标签「机器人」一致」。

**验收标准**

- AC1：未登录用户可答题，但在查看「保存结果到个人画像」或「一键报名推荐项」时引导登录（具体：结果页可展示，**收藏/报名**需登录）。
- AC2：同一用户重复提交问卷，新结果覆盖旧档案并更新时间戳。
- AC3：每个推荐卡片展示分数（可选）+ 不少于 2 条理由（若不足，展示「综合匹配」兜底文案且不报错）。
- AC4：无社团命中极端情况（如新库为空），友好空状态。

---

### 5.4 收藏与对比（P1）

**描述**  
登录用户可收藏社团；在对比页选择 2～3 个社，并列展示关键字段差异；对「招新时间段」做重叠/冲突提示（简单区间重叠算法）。

**验收标准**

- AC1：对比列数 2～3，超出提示替换其中一项。
- AC2：收藏列表与对比状态在刷新后保持（持久化）。

---

### 5.5 报名（P0）

**通用字段（系统级，所有社共享）**

| 字段 | 必填 | 校验 |
|------|------|------|
| 姓名 | ✓ | 2～20 字 |
| 学号 | ✓ | 按学校规则（可配置正则；默认数字 8～12 位） |
| 学院 | ✓ | |
| 年级 | ✓ | 枚举 |
| 手机 | ✓ | CN 11 位 |
| 邮箱 | 选 | 邮箱格式 |
| 自我介绍 | 选 | 上限 500 字 |

**社团自定义题目（P0）**

- 每社最多 **M=5** 道（配置常量）。
- 题型：单行文本 / 多行 / 单选 / 多选。
- 社长可设「是否必填」。

**防重复**

- 同一用户对同一招新季同一社团仅允许 **一条有效报名**（Uniqueness：`userId + clubId + seasonId`）；重复提交提示已报名并跳转「我的报名」。

**状态机（社长可改，新生只读）**

```
submitted → contacted → interview → accepted | waitlist | rejected
```

- P0 至少实现：`submitted`、`contacted`、`accepted`、`rejected`；`interview`、`waitlist` 可并列存在。
- 每次状态变更写入时间轴（谁操作、何时、备注）。

**验收标准**

- AC1：未登录访问 `/apply/:clubId` 跳转登录回跳。
- AC2：招新截止后表单不可提交。
- AC3：社长修改状态后，新生在「我的报名」立即可见（P0 可轮询；P1 WebSocket/通知）。

---

### 5.6 通知（P1）

**触发**

- 报名状态变更。
- 用户收藏社的招新距结束 **24h / 3h**（可配置）提醒（若仍 `recruiting`）。

**载体**

- 站内通知列表 + 未读角标；全部标已读。
- 可选：邮件（二期）。

**验收标准**

- AC1：通知包含跳转 deep link（对应报名记录或社团详情）。
- AC2：用户可在设置中关闭「截止提醒」（可选，P1）。

---

### 5.7 招新专场（P1）

**描述**

- 管理员创建活动：`name`、`date`、`description`、可选 `mapImageUrl`、`boothNumber` 字段存在社团扩展表或专场关联表。
- 首页/专场页按日期列出参展社团。

**验收标准**

- AC1：专场 URL 可分享，未登录可看列表。

---

### 5.8 对话式匹配（P2）

**描述**

- 用户提供自然语言段落（例：「我喜欢打篮球和拍照，一周大概有四个小时，想结交新朋友」）。
- NLU：**规则 + 关键词** 或 调用 LLM API 抽取 `{ interests[], weeklyHours, preferences[] }` 结构体。
- 映射后与 **5.3 相同打分引擎** 接口，输出推荐结果。

**验收标准**

- AC1：抽取失败时 fallback 引导用户改用问卷。
- AC2：展示「根据你的描述，我们识别到…」可解释摘要。

---

### 5.9 运营看板（P2）

**指标**

- 全站 DAU/WAU（若可）、列表 PV、详情 PV、推荐 CTR、报名漏斗（详情→报名页→提交）。
- 分社团排序导出 CSV（可选）。

**验收标准**

- AC1：仅 `admin` 可访问；筛选时间区间。

---

### 5.10 联办活动（P2）

**描述**

- 实体 `JointEvent` 多对多关联社团；展示在各社详情「相关活动」区。

---

### 5.11 社长工作台（P0）

- 编辑社团资料与发布：保存为草稿 / 提交审核（若开启审核流）/ 已发布。
- 配置报名表与自定义题。
- 报名列表：搜索学号/姓名、按状态筛选、批量导出 CSV（P1 可上批量；P0 可单页导出）。
- 数据简版（P0）：近 7 日 PV、报名数、状态分布。

---

### 5.12 管理后台（P0 最小）

- 社团入驻审核：`pending` → `approved` / `rejected`。
- 生成社长邀请码：`INV-XXXX`，一次性或限期。

## 6. 数据模型（逻辑层）

### 6.1 核心实体

**User**

- `id`, `role` (`student`|`leader`|`admin`), `email`, `passwordHash`, `profile`（JSON：姓名等若与报名分离）、`createdAt`

**Club**

- `id`, `name`（unique）, `slogan`, `category`, `tags[]`, `description`, `coverUrl`, `gallery[]`
- `activityTypes[]`, `intensity`, `beginnerFriendly`（bool）
- `recruitStart`, `recruitEnd`, `recruitingComputed`（可由定时任务写缓存字段）
- `contact`（JSON）, `leaderUserId`（主社长）, `status` (`draft`|`pending_review`|`published`|`rejected`)
- `customForm`（JSON Schema 或结构化数组）
- `seasonId`（招新季，可与学年绑定）

**Application**

- `id`, `userId`, `clubId`, `seasonId`
- `commonAnswers`（JSON）, `customAnswers`（JSON）
- `status`, `timeline[]`（事件）
- `createdAt`, `updatedAt`

**Favorite**（P1）  
`userId`, `clubId`, `createdAt`，unique(userId, clubId)

**MatchProfile**（问卷档案）  
`userId`, `answers`（JSON）, `updatedAt`

**Notification**（P1）  
`userId`, `type`, `payload`, `readAt`

**Event** / **ClubEventBooth**（P1）  
专场、展位关联

**JointEvent**（P2）  
`id`, `title`, `clubIds[]`, ...

**AnalyticsEvent**（埋点，可简化）  
`type`, `clubId?`, `userId?`, `sessionId`, `ts`

### 6.2 索引建议

- `Club(status, recruiting)`、`Application(userId, clubId, seasonId)` unique、`Application(clubId, status)`。

| 类别 | 要求 |
|------|------|

| 国际化 | P0 中文即可 |

## 9. 边界与异常

- 社团被下架/删除：详情 404；已报名用户仍可在「我的报名」看到快照（`clubSnapshot` JSON，P0 可仅显示 ID+历史姓名）。
- 问卷/权重配置发布失败：回滚上一版本。
- 并发报名：数据库唯一约束兜底。
